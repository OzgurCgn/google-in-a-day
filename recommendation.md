# Recommendations for Production Next Steps

This project is intentionally designed as a clear, localhost-friendly prototype for a single-machine assignment setting. For production deployment, the first major upgrade should be replacing the current file-based and in-memory work-management approach with more durable infrastructure. A persistent queue such as Redis, RabbitMQ, or Kafka would make interruption recovery much safer and would improve how back pressure is handled under heavier crawl loads. In the same step, moving the primary index and job state from flat files to a proper database such as SQLite or PostgreSQL would provide stronger consistency, safer concurrent access, and easier analytics over stored crawl records.

A second major area of improvement would be crawl governance and ranking quality. In production, I would add robots.txt awareness, configurable same-domain restrictions, stronger URL canonicalization, retry policies with exponential backoff, and clearer handling of external links. On the search side, I would improve ranking beyond the current assignment-focused formula by incorporating document normalization, title weighting, freshness, and possibly BM25-style scoring. Together, these changes would keep the current architecture understandable while making the system more reliable, more responsible, and more scalable in a real deployment environment.
